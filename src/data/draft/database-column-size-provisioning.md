The database contains 2,104 columns that store text data (varchar and nvarchar). Of these, 301 columns are defined as varchar(max) or nvarchar(max), which means they are configured to store unlimited text.

> **VARCHAR** uses a single-byte character set, meaning it can store characters from a single language or script. **NVARCHAR** uses Unicode, allowing it to store characters from multiple languages and scripts.

In the past we have not monitored the use of max columns, because the size of the database has not required us to impose strict limits on them. However, the database has grown large enough that this now poses performance and scalability concerns, especially with the increasing number of active concurrent users. Therefore, we need to begin actively monitoring and limiting the use of max columns in the database.

Here is a brief technical summary of the reasons to avoid varchar(max) when possible:

-   A varchar(max) column cannot be included as a key column in any database indexes.
-   In order to store large text values, varchar(max) columns use LOB\_DATA allocation units, which are much, **much (!)** slower than using IN\_ROW\_DATA storage allocation units.
-   LOB\_DATA storage does not support page and row compression.

Here is a more detailed explanation of the risks and potential problems associated with using varchar(max):

**Performance Issues**:

-   Query Performance: varchar(max) columns negatively impact query performance, especially when dealing with large volumes of data. These columns are stored out-of-row when they exceed 8,000 bytes, requiring additional I/O operations to retrieve the data they contain.
-   Indexing: Indexing varchar(max) columns is problematic because the database cannot fully index them in same way that it can index all other data types, limiting the effectiveness of indexes and slowing the performance of queries that include them.

**Storage Considerations**:

-   Storage Efficiency: Using varchar(max) leads to inefficient storage usage. SQL Server allocates additional space for varchar(max) columns, which may not be used, leading to wasted storage.
-   Data Pages: When varchar(max) data exceeds 8,000 bytes, SQL Server stores the data on separate pages. This separation increases the number of data pages that need to be read during queries, which degrades performance.

**Memory Usage**:

-   Memory Allocation: Queries that include varchar(max) columns consume more memory, impacting overall system performance. SQL Server may need to allocate larger buffers to handle varchar(max) columns, reducing the memory available for other operations.

**Data Integrity and Maintenance**:

-   Data Validation: Defining specific lengths for varchar columns helps to enforce data integrity and validation. varchar(max) lacks constraints, making it harder to ensure consistent data entry.
-   Maintenance: Managing and maintaining databases with varchar(max) columns is more challenging. Tasks such as backups, restores, and index maintenance take longer and require more resources.
-   Efficiency: Limiting text based on the expected size of the data helps to optimize storage and performance. Specifying a reasonable length for varchar columns allows SQL Server to allocate storage more efficiently and better optimize its queries on stored data values.

## Provisioning Best Practices

-   Use varchar(max) only when the length of a single value is expected to exceed 8K (i.e., 8000 characters). Otherwise, use nvarchar(limit) where "limit" is a reasonable constraint on the expected size of the values to be stored in the column
-   Use nvarchar only when you need to store characters from multiple languages and character sets.
-   Use nvarchar(max) only when the length of a single value is expected to exceed 4K (i.e., 4000 characters). Otherwise, use nvarchar(limit) where "limit" is a reasonable limit on the expected size of the values to be stored in the column.
-   Never define multiple varchar(max) columns in the same database table. If there is a legitimate system requirement to store more than 8K in a varchar column (or more than 4K in an nvarchar column) then your schema should be designed so that no database table contains more than one such column.
-   When using varchar(limit) and nvarchar(limit), set a realistic limit that is based on expected usage. The SQL Server query optimizer expects values in such a column have an average length that is limit \* 0.5, therefore it is important to select a realistic limit for performance reasons, in addition to data integrity reasons.
-   A column that stores a single email address must be varchar(254). Refer to [https://www.directedignorance.com/blog/maximum-length-of-email-address](https://www.directedignorance.com/blog/maximum-length-of-email-address)
-   A column that stores a single URL must be varchar(2000). Refer to [https://stackoverflow.com/questions/417142/what-is-the-maximum-length-of-a-url-in-different-browsers](https://stackoverflow.com/questions/417142/what-is-the-maximum-length-of-a-url-in-different-browsers)

## Immediate Next Steps

1.  Analyze the existing data in every database table, including all five enterprise partitions (E01-E05).
2.  For every varchar column, determine the length of the largest value currently stored in the database.
3.  Calculate an optimum limit based on the largest existing value:
    1.  Round up to the nearest 100 characters.
    2.  Add 20 percent.
4.  If the optimum limit is less than 8K then replace varchar(max) with varchar(limit) using the calculated optimum. Otherwise, leave the column defined as varchar(max).
5.  Repeat steps 2, 3, and 4 for every nvarchar column and use a 4K threshold.
6.  Ensure every column that stores an email address is varchar(254).
7.  Ensure every column that stores a URL is varchar(2000).
8.  Write a SQL Upgrade script to update the database schema with the new limits.
9.  Update the Entity Configuration classes in the data access layer with the new limits.
10.  Update any MaxLength attribute values in the UI with the new limits.
